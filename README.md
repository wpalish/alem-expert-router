# Alem Expert Router

**Умное распределение экспертов, менторов и фрилансеров по входящим заявкам** —
с учётом компетенций, загрузки и приоритетов. Рабочий прототип под задачу
Alem School «Автоматизация и оптимизация внутренних процессов».

> Проблема: нет удобного и прозрачного механизма управления заявками и подбора
> исполнителей. Решение: API, который автоматически и **объяснимо** назначает
> лучшего свободного эксперта на каждую заявку.

## Что решает (1:1 с постановкой задачи)

| Требование заказчика | Как закрывается |
|---|---|
| Управление заявками и объявлениями | REST API приёма заявок и пула исполнителей |
| Автоподбор по **компетенциям** | строгая проверка покрытия навыков + оценка глубины владения |
| Учёт **загрузки** | балансировка: ёмкость уменьшается «на лету», никто не перегружен |
| Учёт **приоритетов** | заявки обрабатываются по приоритету, затем по сроку |
| **Прозрачность** управления ресурсами | каждое назначение с объяснением + дашборд загрузки |
| Без роста операционных затрат | полностью автоматический прогон, ноль ручного труда |

## Как работает алгоритм

1. **Жёсткие фильтры** — эксперт недоступен / нет свободной ёмкости / не
   покрывает требуемые навыки → отсеивается.
2. **Оценка 0..100** среди подходящих = взвешенная сумма: компетентность (0.6) +
   балансировка загрузки (0.25) + рейтинг (0.15). Веса настраиваются под
   политику школы без правки кода.
3. **Распределение по приоритету**: высокоприоритетные заявки получают лучших
   свободных исполнителей первыми; ёмкость списывается по ходу.

Каждое назначение возвращает список причин — почему выбран именно этот человек.

## Запуск

```bash
python -m venv .venv && source .venv/bin/activate
pip install -r requirements.txt

# тесты (доказательство корректности)
pytest -q

# API
uvicorn app.main:app --reload
# Swagger UI: http://localhost:8000/docs
```

Быстрый сценарий:

```bash
curl -X POST localhost:8000/seed       # загрузить демо-данные
curl -X POST localhost:8000/assign     # умное распределение
curl localhost:8000/dashboard          # прозрачная картина загрузки
```

## Возможности

- **Веб-приложение** с вкладками: Распределение · Заявки · Эксперты · Объявления
  (формы, CRUD, удаление, метрики эффекта).
- **Глобально-оптимальное распределение** (венгерский алгоритм), а не жадный перебор.
- **Частичное покрытие**: если точного исполнителя нет, система предлагает
  «почти подходящего» (со штрафом и пометкой), вместо того чтобы оставить заявку без ответа.
- **Ручное назначение** (`POST /assign/manual`): координатор закрепляет исполнителя
  вручную — это перебивает авто-распределение для конкретной заявки.
- **Объявления** (`/announcements`): поиск экспертов, доступные задачи, мероприятия —
  вторая часть постановки «управление заявками и объявлениями».
- **AI-слой**: заявку можно описать свободным текстом — система извлекает
  компетенции (`/requests/parse`) с учётом отрицаний. LLM через Groq при наличии
  ключа, иначе — детерминированный fallback по таксономии (работает без ключей).
- **REST-семантика**: `POST` создаёт (id генерируется или 409 при дубле), `PUT` обновляет.
- **Persistence**: данные в SQLite (WAL), переживают перезапуск.

## Как именно распределяются заявки (и почему не жадно)

Наивный подход «каждой заявке по очереди — лучший свободный эксперт» **не даёт
глобального оптимума**: локально лучший выбор для одной заявки может оставить
другую без исполнителя. Пример: заявке A нужен `python`, заявке B — `python`+`ml`;
эксперт X владеет обоими, Y — только `python`. Жадность отдаст X заявке A (выше
оценка) и оставит B ни с чем. Правильно: A→Y, B→X — закрыты обе.

Поэтому распределение сведено к **задаче о назначениях** и решается венгерским
алгоритмом (Кун–Манкрес) на матрице «заявка × слот исполнителя». Эксперт с
ёмкостью N даёт N слотов. Вклад каждой заявки взвешен на приоритет, поэтому при
конкуренции за один слот выигрывает более важная заявка; небольшой штраф за
концентрацию разносит равноценные заявки по разным людям. Решатель работает на
чистом Python без зависимостей; если установлен SciPy — используется его
`linear_sum_assignment`.

## Ограничения и компромиссы (честно)

- **Приоритет — это вес, а не жёсткий лок-степ.** Очень важная заявка получает
  повышенный вклад, но теоретически система может пожертвовать одной
  высокоприоритетной заявкой ради двух средних, если суммарная выгода выше. Это
  осознанный выбор «максимум пользы»; при необходимости строгий приоритет
  делается лексикографически (сначала закрыть все P5, потом P4 …).
- **AI-fallback — это таксономия ключевых слов с учётом отрицаний**, а не
  семантика. Полноценный разбор — через LLM (Groq/OpenAI-совместимый) при ключе.
- **Живое демо на GitHub Pages** повторяет логику на JS в браузере (без сети);
  источник истины для алгоритма — Python-бэкенд и его тесты.

## Дорожная карта (MVP → продукт)

- **MVP (есть сейчас):** движок матчинга + REST API + дашборд + тесты.
- **Этап 2:** PostgreSQL, аутентификация (роли: студент / координатор / ментор),
  веб-кабинет подачи заявок, уведомления.
- **Этап 3 (AI-слой):** семантический разбор свободного текста заявки в
  компетенции через LLM/эмбеддинги (заявку можно писать словами — система сама
  извлечёт нужные навыки), прогноз сроков, авто-ребалансировка нагрузки.
